# Studio Readiness Checklists

Lightweight self-assessment tools for each training phase.

**These are mirrors, not gates.** They help you answer: "Am I actually ready for the next kind of decision?" They should never block actions.

---

## How to Use These

| Context | How |
|---------|-----|
| **In-app** | Expandable panel in settings or help |
| **Training** | Reference during onboarding sessions |
| **Print** | One-page handout per phase |
| **Self-check** | Quick review before major changes |

---

## Phase 1: Onboarding

**Goal: Operational Trust**

### Checklist: "Can We Run Tomorrow?"

#### Studio Basics
- [ ] Studio profile created (name, timezone, contact info)
- [ ] At least one location defined (physical or virtual)

#### Offerings & Schedule
- [ ] Core offerings defined (classes, events, trainings)
- [ ] Sessions scheduled at least 1–2 weeks out
- [ ] Capacity limits reviewed and understood

#### People & Roles
- [ ] Owner/admin role confirmed
- [ ] Instructors added with correct permissions
- [ ] Staff understand where to see schedules and rosters

#### Bookings & Attendance
- [ ] A test booking completed successfully
- [ ] Roster visible to staff
- [ ] Attendance can be marked without confusion

#### Confidence Check
- [ ] "If I step away for a day, nothing breaks"
- [ ] "Staff could answer basic questions without me"

### Exit Signal

> You are no longer actively thinking about the software during normal operations.

---

## Phase 2: Branding

**Goal: Ownership & Confidence**

### Checklist: "Does This Feel Like Us?"

#### Internal App (Light Branding Only)
- [ ] Logo applied
- [ ] Brand colors visible but subtle
- [ ] Internal views still feel familiar and readable

#### Consumer-Facing Experiences
- [ ] Booking pages reflect studio tone and imagery
- [ ] Event or training pages use brand voice
- [ ] Visual hierarchy feels intentional, not cluttered

#### Boundary Awareness
- [ ] No internal workflows redesigned
- [ ] No staff-facing layouts altered for aesthetics
- [ ] Accessibility and readability preserved

#### Collaboration Readiness
- [ ] Designer or agency understands "consumer-facing only"
- [ ] Branding guide shared with collaborators
- [ ] Changes reviewed before publishing

### Exit Signal

> Clients recognize the brand. Staff didn't need retraining.

---

## Phase 3: Growth

**Goal: Operational Leverage**

### Checklist: "Are We Scaling Intentionally?"

#### Visibility & Insight
- [ ] Reports answer real business questions
- [ ] Exports are used (not just available)
- [ ] Data flows cleanly to accounting or ops tools

#### Process Leverage
- [ ] Repetitive manual work identified
- [ ] Integrations added only where they remove effort
- [ ] No automation added "just because"

#### Delegation
- [ ] Staff can manage sessions independently
- [ ] Owners are no longer a bottleneck
- [ ] Errors are recoverable without panic

#### Complexity Check
- [ ] "This feels easier, not heavier"
- [ ] "If we paused growth, the system still makes sense"

### Exit Signal

> The studio does more with less attention.

## Re-Entry Checklist

Use when something significant changes:

| Trigger | Action |
|---------|--------|
| New location added | Revisit Phase 1 for that location |
| New program type launched | Revisit Phase 1 → 2 |
| Major staff turnover | Revisit Phase 1 |
| Significant branding refresh | Revisit Phase 2 |

**Quick re-entry process:**
1. Briefly review Phase 1 checklist
2. Confirm Phase 2 still applies
3. Assess if Phase 3 adjustments needed

This prevents cascading mistakes.

## Integration Points

### For In-App Implementation

```
Location: /manage/settings/readiness
Display: Collapsible accordion per phase
State: Checkbox progress stored per studio
Behavior: Visual progress indicator, never blocking
```

### For Training Materials

Embed as:
- PDF handout
- Slide deck appendix
- Video chapter markers

### For Support

Reference when:
- Studio reports confusion → check Phase 1
- Brand changes cause issues → check Phase 2
- Growth feels painful → check Phase 3
